# Hundred and Ten Web Frontend

## Stack

- Vite 8 + TanStack Router (file-based routing), React 19, TypeScript, Tailwind CSS v4
- Firebase auth (client-side only, Google sign-in)
- Fully static SPA — deployed to Azure Static Web Apps

## Project Structure

- `routes/` — TanStack Router file-based route files; `routeTree.gen.ts` auto-generated by the plugin (committed)
- `app/` — page-level components (game-page, lobby-detail) and `globals.css`
- `components/` for shared UI components
- `lib/` for API client, hooks, and utilities
- `main.tsx` at repo root — app entry point
- `index.html` at repo root — HTML shell
- `docs/solutions/` — documented solutions to past problems, organized by category with YAML frontmatter (`module`, `tags`, `problem_type`)
- Path alias: `@/*` maps to `./*` (repo root, not `src/`)

## Commands

```bash
npm run dev        # Start Vite dev server
npm run build      # tsc -b && vite build (outputs to dist/)
npm run preview    # Preview the dist/ bundle locally
npm run lint       # ESLint check
npm run lint:fix   # ESLint autofix
npm run clean      # Remove dist/ and out/ directories
```

## Conventions

- Pure SPA — no SSR, no `"use client"` directives (meaningless in Vite; do not add them)
- API base URL from `VITE_API_URL` env var accessed via `import.meta.env.VITE_API_URL` (never hardcoded)
- Auth tokens managed in-memory by Firebase SDK
- Tailwind v4 configured via CSS (`@import 'tailwindcss'` + `@theme` in `app/globals.css`), not `tailwind.config.js`
- Tailwind uses `@tailwindcss/vite` plugin in `vite.config.ts` (no `postcss.config.js`)
- ESLint 9 flat config with prettier
- Mobile-first design — 44px minimum touch targets, horizontal scroll for card hand
- `routeTree.gen.ts` is auto-generated — if you add a new route file, run `npm run dev` or `npm run build` to regenerate it before committing

## API Patterns

- Backend at `VITE_API_URL`, all endpoints scoped under `/players/{player_id}/...`
- Search endpoints are POST with JSON body `{ searchText, offset, limit }`
- Action/start endpoints return `Event[]`, not updated game — must re-fetch after
- Backend enforces all authorization; frontend role checks are cosmetic

## Routing

- TanStack Router file-based routing in `routes/` directory
- Route file naming: `routes/__root.tsx` (root layout), `routes/index.tsx` (`/`), `routes/lobbies/index.tsx` (`/lobbies`), `routes/lobbies/new.tsx`, `routes/lobbies/$lobbyId.tsx`, `routes/games/$gameId.tsx`
- Use `useNavigate()` + `navigate({ to: path })` instead of `router.push()`
- Use `useParams({ from: '/route/$param' })` for type-safe route params
- Use `<Link to="...">` from `@tanstack/react-router` (not `next/link`)

## Environment Variables

- All env vars use `VITE_` prefix: `VITE_API_URL`, `VITE_FIREBASE_API_KEY`, `VITE_FIREBASE_AUTH_DOMAIN`, `VITE_FIREBASE_PROJECT_ID`
- Access pattern: `import.meta.env.VITE_*`
- In Vitest tests: use `vi.stubEnv('VITE_*', value)` / `vi.unstubAllEnvs()`

## Testing

- Unit tests required for API client (`lib/api/client.ts`) and polling hook (`lib/hooks/use-polling.ts`)
- Type checking: `npx tsc --noEmit`
- Build verification: `npm run build` must produce `dist/` directory with `index.html` and `staticwebapp.config.json`

## Operational Notes

- After deployment to Azure Static Web Apps, add the deployment URL to Firebase Console authorized domains (Firebase Console → Authentication → Settings → Authorized domains). Without this, Google sign-in fails with `auth/unauthorized-domain`.
- `VITE_*` env vars are inlined into the client bundle at build time — Firebase config values are visible in the built JS. This is expected; Firebase API keys are not secrets (security is enforced by Firebase security rules).
- Azure Static Web Apps SPA fallback is configured in `public/staticwebapp.config.json` — all navigations fall back to `index.html` for client-side routing.
